# Design Draft: Multiple Voices for Dialogues

## Status
**DRAFT** - Brainstorming Phase (Not yet implemented)

## Problem Statement
Currently, all dialogues in the Armenian Learning Assistant are generated using a single voice via Piper TTS. While we deduplicate `Phrase` text, this limits the realism of dialogues (which often feature 2 or more speakers). 
If a dialogue has multiple speakers, they should ideally sound distinct. However, generating an infinite number of voices is technically and economically unfeasible, and blindly generating new audio for every speaker breaks our media deduplication goals.

## Proposed Architecture

### 1. Voice Identifiers
- Treat a `Phrase` as a hub for its audio representation. A single `Phrase` can have multiple `Media` objects attached to it.
- Each `Media` object has a `voice_identifier` (e.g., `uuid-voice-1`, `uuid-voice-2`, or named descriptors like `hy-AM-Aram`).

### 2. Dialogue Speakers & Voice Assignment
- `DialogueSpeaker` will hold a `voiceIdentifier`.
- During dialogue generation:
  1. Determine how many unique speakers are in the dialogue.
  2. Assign a specific `voiceIdentifier` to each `DialogueSpeaker`.
  
### 3. Smart Deduplication & Voice Reuse
Instead of always triggering new TTS generation, the system will use a smart fallback/reuse strategy:
- When iterating through the generated `DialoguePhrase`s, check the underlying `Phrase` entities.
- If the `Phrase` already has a `Media` object matching the assigned `voiceIdentifier`, **reuse it**.
- **Cross-Phrase Reuse Strategy:** Look at all phrases assigned to a speaker in the dialogue. If the underlying `Phrase`s already share a specific `voiceIdentifier` (e.g., they all already have audio generated by "Voice B"), tentatively assign "Voice B" to that speaker to maximize reuse and minimize TTS calls.
- If we already have 2 or 3 voices available among the phrases of the dialogue, we map those existing voices to the current dialogue speakers rather than picking an entirely new unused voice, greatly optimizing storage and TTS processing.
- Only trigger new TTS generation if a phrase absolutely lacks audio for the assigned voice.

## Maximum Reasonable Limit for Speakers
- **Sweet Spot:** 2 to 3 speakers per dialogue (e.g., A/B dialogue, or Narrator + 2 characters).
- **Total System Limit:** 4 to 5 distinct base voices for the entire application is more than sufficient for 99% of language learning scenarios.
- By capping the total number of distinct voices at ~5, we ensure that over time, the most common phrases (e.g., "Hello", "Yes", "No") will cache all 5 audio variants, drastically speeding up dialogue generation without blowing up storage.

## Future Exploration (Insane Mode)
- Utilizing open-source datasets (like Mozilla Common Voice, CC0 licensed) to train custom Piper (VITS) models locally.
- Hiring voice actors for one-off gigs to legally own 2-3 custom Armenian voice models.
- Abstracting voice models behind UUIDs allows hot-swapping the underlying `.onnx` model files without database migrations.

## API Constraints & LLM Integration

### 1. Jakarta Validation for Speaker Limits
To enforce the maximum limit of 5 speakers, we must add `@Size(max = 5)` (or a similar constraint) to the relevant DTOs:
- **`POST /chat/dialogue`**: Validate the generation resource (`com.blbulyandavbulyan.larm.ai.chat.StructuredDialogueResource`) so the user/system cannot request more than 5 speakers.
- **`POST /dialogues`**: Validate the save request to prevent saving any dialogue with more than 5 speakers.

### 2. LLM Transparency & Explanations (`DialogueChatResponse`)
The `message` field description in the `DialogueChatResponse` (or equivalent schema provided to the LLM) should explicitly instruct the AI:
- "If you were forced to alter the user's original request due to system constraints (e.g., the user asked for 7 speakers but you only generated 5 because of the maximum limit), you MUST use this message field to explain to the user why their request was modified."
- This prompt instruction should apply to all Jakarta constraints generally, ensuring the AI behaves transparently when hitting system limits.

### 3. Security: Preventing Stack Trace & Exception Leaks
When Jakarta validation fails (either during user input or LLM generation), we must ensure internal Java architecture is not leaked to the LLM prompt or the end user.
- **`JakartaValidationAdvisor`**: This advisor intercepts LLM outputs and feeds validation errors back into the prompt. We must guarantee that the `violationsMessage` construction strictly uses `v.getPropertyPath() + " " + v.getMessage()`. It must *never* perform a `.toString()` on the raw validated object or the exception itself, as that would leak Java class names directly into the prompt.
